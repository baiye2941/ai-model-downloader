| 全局状态重构 | **已完成** | `AppState` 结构体 + Tauri `manage()` 注入 |

### 2.2 仍需的架构优化

#### 2.2.1 消除剩余重复定义

| 重复项 | 出现次数 | 解决方案 |
|--------|---------|---------|
| `FragmentRecord` | 2 处 (fragment.rs, orchestrator.rs) | 统一为一个定义 |
| `Storage` vs `AsyncStorage` trait | 2 处 (qf-core, qf-io) | 统一为 `qf-io::AsyncStorage`，qf-core 仅 re-export |
| `extract_filename_from_path` | FTP 独立实现 (ftp.rs:338) | 复用 qf-core 的 `extract_filename_from_url` |

#### 2.2.2 Protocol trait 流式化

当前 `Protocol::download_range` 返回 `Bytes`，整个范围加载到内存，违背零拷贝目标。

**方案**：改为流式接口：

```rust
fn download_range(&self, url: &str, start: u64, end: u64) 
    -> impl Future<Output = QfResult<impl AsyncRead>> + Send;
```

分片下载时用 `tokio::io::copy_buf` 流式写入存储，内存占用恒定。

#### 2.2.3 错误体系增强

- `QfError::Other(String)` 丢失原始错误类型 → 改为 `QfError::Other(Box<dyn Error + Send + Sync>)`
- Tauri 命令返回 `Result<T, String>` → 改为结构化错误类型 `AppError`，实现 `Serialize`
- `Verifier::verify` 返回 `QfResult<bool>` → 改为 `QfResult<()>`，不匹配时返回 `ChecksumMismatch`

#### 2.2.4 持久化层

当前所有状态纯内存，重启丢失。需要新增：

- `qf-store` crate：基于 `sled` 或 `redb` 的嵌入式 KV 存储
- 持久化：任务状态、配置、DHT 节点、Peer 评分
- 启动时恢复未完成任务（断点续传基础）

#### 2.2.5 缺失的 Kademlia DHT 核心逻辑

当前 `KademliaDht` 仅有节点存储，缺少：
- XOR 距离度量
- k-bucket 路由表
- FIND_NODE / FIND_VALUE / STORE 查询
- 迭代查找算法
- 节点 ID 生成（当前接受任意 ID）

#### 2.2.6 调度器与引擎集成

当前 `qf-scheduler` 和 `qf-engine` 独立运行，缺少：
- 调度器根据带宽预测动态调整引擎并发数
- 调度器驱动分片重排（rarest-first 策略）
- 任务完成度提升优先级的实时反馈

#### 2.2.7 qf-app 真实下载管线

当前 `task_fn` 模拟下载（`sleep(2ms)` + chunk 计数），需替换为：
- 调用 `DownloadTask::execute()` 真实下载管线
- 通过 Protocol trait 发起真实网络请求
- 集成 WritePipeline 流式写入

## 三、算法优化

### 3.1 带宽预测器

| 当前 | 优化 |
|------|------|
| Holt-Winters 双指数平滑 | 增加季节性分量（三指数平滑），捕捉日间带宽波动 |
| 固定 alpha/beta | 自适应参数：根据预测误差自动调整平滑系数 |
| `confidence()` 每次调用计算 `exp()` | 查表法或多项式近似 |
| `observe()` 接受 `f64`，与 `BandwidthTracker` 的 `u64` 不匹配 | 统一类型，添加适配器 |

### 3.2 分片策略

| 当前 | 优化 |
|------|------|
| 固定 min/max 分片大小 | 根据网络抖动（jitter）动态调整：低抖动用大分片，高抖动用小分片 |
| EWMA 单一 alpha | 多时间尺度 EWMA（短/中/长期），类似 TCP 的 RTT 估算 |
| 分片数固定 | 超分片引擎：初始保守分片，根据实时带宽反馈逐步增加并发 |

### 3.3 源选择算法

| 当前 | 优化 |
|------|------|
| 固定权重 (0.3/0.4/0.2/0.1) | 可配置权重 + 基于贝叶斯更新的动态权重 |
| 100Mbps 归一化上限 | 自适应归一化：基于观察到的最大带宽 |
| Xorshift64 全局 PRNG | 改用 per-thread `ChaCha8Rng`，消除全局原子竞争 |
| 加权随机选择 | 增加 ε-greedy 探索：90% 利用最优源，10% 探索新源 |

### 3.4 连接池

| 当前 | 优化 |
|------|------|
| 全局 + per-host 信号量 | 增加 per-IP 段限制，防止单 CDN 域名多 IP 绕过限制 |
| `entry(host.to_string())` 每次分配 | 使用 `Arc<str>` 或 `raw_entry` API 避免重复分配 |
| 无连接复用 | HTTP keep-alive 连接池（reqwest 已内置，需正确配置） |

### 3.5 DHT 节点淘汰

| 当前 | 优化 |
|------|------|
| O(n) 遍历找最旧节点 | 使用 `BTreeMap<Instant, NodeId>` 或 `IndexMap` 维护插入顺序 |
| `Instant` 不可序列化 | 改用 `chrono::DateTime` 或 `SystemTime` |

## 四、I/O 层优化

### 4.1 io_uring 实现（4 个 `todo!()`）

需完成：
- `submit_write`（`iouring.rs:294`）：构建 `Write64` SQE，提交到提交队列
- `read_at`（`iouring.rs:332`）：构建 `ReadFixed` SQE，使用预注册 fixed buffer 实现零拷贝
- `sync`（`iouring.rs:352`）：构建 `Fsync` SQE
- `allocate`（`iouring.rs:373`）：使用 `fallocate` 或 io_uring `Fallocate` 操作

buffer 对齐修复：使用 `aligned_alloc` 或 `Vec::with_capacity` + 手动对齐。

### 4.2 WritePipeline 集成

当前 `BufferPool` 在 `WritePipeline` 中未使用。应实现：
- 从 `BufferPool` 获取 buffer
- 网络数据读入 pool buffer
- 提交 buffer 给存储层写入
- 写入完成后释放 buffer 回 pool

### 4.3 TokioFile 优化

- 缓存文件游标位置，顺序写入时跳过 seek
- 大文件使用 `pread`/`pwrite`（通过 `tokio-uring` 或 `nix`）避免 seek 开销
- 添加写入超时保护

### 4.4 Windows 优化

- `WinFile` NO_BUFFERING 模式必须强制 512/4096 字节对齐
- 使用 `FILE_FLAG_WRITE_THROUGH` 替代 `fsync` 保证持久性
- 考虑 Windows `FILE_FLAG_OVERLAPPED` + IOCP 异步 I/O

### 4.5 Kernel Bypass（Linux only，新增）

- 使用 AF_XDP socket 旁路内核网络栈
- 适用场景：高吞吐量下载（> 1Gbps）
- 通过 feature flag `kernel-bypass` 控制编译（默认关闭）
- 回退策略：不支持时自动降级到标准 socket

### 5.4 嗅探层：双模式资源嗅探

**方案**：Tauri WebView 拦截 + 浏览器扩展，双模式组合覆盖所有浏览场景。

#### 模式一：Tauri WebView 拦截（应用内浏览）

零额外依赖，覆盖用户在 QuantumFetch 内嵌浏览器标签页中的浏览行为。

```
用户浏览网页(应用内 BrowserTab)
  → Tauri WebViewBuilder / wry RequestHandler 拦截响应头
  → 检测 Content-Type / Content-Disposition / Content-Length
  → 匹配可下载资源类型 → 存入 ResourceManager
  → Tauri 事件 `sniffer:resource-captured` 推送前端
  → 前端 SnifferPanel 实时展示，用户一键下载
```

**Tauri v2 API**：

- `WebViewBuilder::on_navigation()` — 拦截导航请求
- 自定义协议 (`tauri::protocol`) — 拦截所有请求/响应
- `wry` 层的 `RequestHandler` — 底层请求拦截（最灵活）

**实现要点**：

1. 新增 `BrowserTab` 组件：应用内嵌入 WebView 标签页
2. 注册请求拦截器：在响应阶段检查 Content-Type 和 Content-Length
3. 资源匹配规则复用 `qf-sniffer/capture.rs` 的 `identify_resource()` + `should_capture()`
4. 捕获的资源通过 Tauri 事件推送到前端

**优势**：零额外依赖、无需安装扩展、无需系统代理、与 Tauri 天然集成。

**局限**：仅覆盖应用内浏览，用户在系统浏览器中上网时无法嗅探。

#### 模式二：浏览器扩展（系统浏览器嗅探）

覆盖用户日常使用 Chrome/Firefox/Edge 的所有浏览行为，达到 IDM 同等体验——**零行为改变**。

```
用户正常使用系统浏览器
  → 扩展 background.js 通过 webRequest API 拦截响应头
  → 匹配 Content-Type / Content-Length / 扩展名
  → 通过 Native Messaging 发送资源信息到 qf-app
  → qf-app 存入 ResourceManager + Tauri 事件推送前端
  → SnifferPanel 实时展示 / 系统通知提示
```

**扩展结构**：

```
qf-sniffer-extension/
  manifest.json          # 权限: webRequest, nativeMessaging, downloads
  background.js          # webRequest.onHeadersReceived 拦截响应头
  content.js             # 可选: 页面内浮动按钮，快速下载当前页资源
  native_host/            # Native Messaging Host
    qf_sniffer.json      # 清单文件
    qf_sniffer_host      # 可执行文件，转发到 qf-app Unix socket / Named Pipe
```

**关键技术**：

- `chrome.webRequest.onHeadersReceived` — 拦截所有响应头，无需拦截请求体，零性能开销
- Native Messaging — 扩展通过 stdin/stdout 与本地 Host 通信，Host 转发到 qf-app
- 跨浏览器 — Chrome/Edge 用同一套(Chromium)，Firefox 需适配 `browser.*` API

**实现要点**：

1. `manifest.json`：声明 `webRequest`、`nativeMessaging` 权限和匹配规则
2. `background.js`：约 50-80 行，注册 `onHeadersReceived` 监听器，匹配可下载资源，通过 Native Messaging 发送
3. Native Messaging Host：轻量可执行文件，与 qf-app 通过 IPC（Unix socket / Windows Named Pipe）通信
4. qf-app 端：新增 IPC 监听服务，接收扩展推送的资源信息
5. 安装程序：自动注册 Native Messaging Host 到系统浏览器

**优势**：覆盖系统浏览器全部流量、用户零行为改变、与 IDM 体验一致。

**局限**：需安装扩展、需开发多浏览器适配、需 Native Messaging Host 注册。

#### 双模式协同

| | 应用内 WebView | 系统浏览器扩展 |
|---|---|---|
| 覆盖场景 | QuantumFetch 内浏览 | 日常使用 Chrome/Firefox/Edge |
| 依赖 | 零 | 需安装扩展 |
| 用户感知 | 需切换到应用内 | 零行为改变 |
| 数据流 | Tauri 事件直接推送 | Native Messaging → IPC → Tauri 事件 |
| 优先级 | 内置，开箱即用 | 可选安装，后续补充 |

两种模式捕获的资源统一汇入 `ResourceManager`，前端 SnifferPanel 无感知来源差异。

## 十一、下载管线瓶颈专项（1Gbps+ 吞吐量关键）

> 以下问题直接影响 1Gbps+ 吞吐量，是超越 IDM 的核心瓶颈。

### 11.1 文件 I/O 串行化——最关键瓶颈

**现状**：`TokioFile` 和 `WinFile` 均使用 `tokio::sync::Mutex<tokio::fs::File>` 串行化所有操作。每个 `write_at`/`read_at`/`sync` 都需要：获取锁 → seek → I/O → 释放锁。

16 并发分片 × 64MB 分片 × 1Gbps = ~1600 次 write_at/秒。每次 = lock + seek(syscall) + write(syscall) + unlock ≈ 4 次上下文切换。总计 6400 次上下文切换/秒。

**目标**：消除 seek 和锁竞争，允许多分片并发写入不同偏移。

**方案**：

| 平台 | 机制 | 说明 |
|------|------|------|
| Linux | `pread`/`pwrite` via `tokio-uring` | 原生定位 I/O + io_uring 提交，零锁零 seek |
| Windows | `FILE_FLAG_OVERLAPPED` + IOCP | OVERLAPPED.Offset 指定写入位置，内核级并发 |
| macOS | `pread`/`pwrite` via `nix` | 同 Linux 方案 |
| 通用回退 | `RwLock` 替换 `Mutex` | 至少允许并发 `file_size()` 读取 |

**实现路径**：
1. `AsyncStorage` trait 新增 `write_at_positioned(&self, offset: u64, data: &[u8]) -> QfResult<usize>`
2. `TokioFile` 实现：Linux 用 `nix::sys::pread`，Windows 用 OVERLAPPED
3. `WinFile` 实现：`WriteFile` + `OVERLAPPED{Offset, OffsetHigh}`
4. 移除 `Mutex<tokio::fs::File>`，改为 `Arc<tokio::fs::File>` 或 `OwnedFd`

### 11.2 网络到磁盘无流式管道

**现状**：`execute_fragmented_download` 中每个分片先 `download_range` 获取完整 `Bytes`，再一次性 `write_at` 写入。

峰值内存 = max_concurrent_fragments × max_fragment_size = 16 × 64MB = **1GB**。

网络下载和磁盘写入完全串行——下载完一个分片才开始写，写入时网络空闲。

**目标**：分片内部也流式传输，网络和磁盘 I/O 重叠。

**方案**：

```rust
// 流式分片下载伪代码
async fn download_fragment_streaming(
    protocol: &dyn Protocol,
    storage: &dyn AsyncStorage,
    url: &str, offset: u64, size: u64,
    buf_pool: &BufferPool,
) -> QfResult<()> {
    let stream = protocol.download_range_stream(url, offset, offset + size - 1).await?;
    let mut pos = offset;
    let mut buf = buf_pool.alloc().await;
    pin_mut!(stream);
    loop {
        match stream.next().await {
            Some(chunk) => {
                buf.extend_from_slice(&chunk);
                if buf.len() >= buf_pool.chunk_size() {
                    storage.write_at(pos, &buf).await?;
                    pos += buf.len() as u64;
                    buf.clear();
                }
            }
            None => {
                if !buf.is_empty() { storage.write_at(pos, &buf).await?; }
                break;
            }
        }
    }
    buf_pool.release(buf);
    Ok(())
}
```

**效果**：峰值内存 = max_concurrent × chunk_size（16 × 1MB = 16MB，降低 60 倍）。网络和磁盘 I/O 管道化，吞吐量提升 30-50%。

### 11.3 进度更新锁竞争

**现状**：`task_fn` 每个模拟 chunk（100KB，2ms 间隔）锁 `tasks` 5 次。真实下载中 chunk 更小、频率更高。

1Gbps / 100KB = 1250 chunks/秒 × 5 lock = 6250 次 lock/秒/任务。前端 `get_task_list()` 也在锁 `tasks`，级联阻塞。

**方案**：用 `tokio::sync::watch` 通道替代直接锁操作。

```rust
struct AppState {
    tasks: Mutex<HashMap<String, TaskInfo>>,
    progress_tx: watch::Sender<HashMap<String, TaskProgress>>,
    progress_rx: watch::Receiver<HashMap<String, TaskProgress>>,
}
```

下载循环仅修改本地进度，每 N 毫秒批量 `send` 一次。前端 `watch::Receiver::changed()` 等待更新，无需轮询。

### 11.4 信号量顺序等待阻塞

**现状**：`for frag in &fragments` 循环先 `acquire_owned().await` 再 `tokio::spawn`。信号量满时循环阻塞。

**方案**：先 spawn 所有任务，在任务内部 acquire 信号量。

```rust
for frag in &self.fragments {
    let handle = tokio::spawn(async move {
        let _permit = semaphore.acquire().await?;
        // 下载分片...
    });
    handles.push(handle);
}
```

### 11.5 FragmentRecord.data 字段违背零拷贝

**现状**：`FragmentRecord.data: Option<Bytes>` 可持有 64MB 数据，但真实管线从不设置此字段。该字段仅在 `complete_download()` 中使用，而此函数从未被真实管线调用。

**方案**：移除 `FragmentRecord.data` 字段。分片数据流经 BufferPool → Storage，不应驻留在分片记录中。

### 11.6 verify() 每分片分配大缓冲区

**现状**：`let mut buf = vec![0u8; frag.info.size as usize]`（`downloader.rs:513`），64MB 分片 = 64MB 堆分配。

**方案**：使用 BufferPool 分配，或按固定 chunk 大小（1MB）分块读取校验。

### 11.7 分片查找 O(n) 线性搜索

**现状**：`self.fragments.iter_mut().find(|f| f.info.index == index)`（`downloader.rs:484`），100+ 分片时每次完成做 O(n) 扫描。

**方案**：新增 `HashMap<u32, usize>` 索引，或改用 `Vec` 按 index 顺序存储以 O(1) 访问。

### 11.8 WritePipeline 批量写入未合并

**现状**：`write_batch` 对多个 segment 逐个写入再 sync。相邻 segment 可合并为一次 I/O。

**方案**：检测相邻 segment，合并为一次 `write_at` 调用。减少 syscall 从 N+1 到 1+1。

## 十二、并发模式优化

### 12.1 Mutex → RwLock / DashMap / lock-free

| 位置 | 当前 | 问题 | 目标 |
|------|------|------|------|
| `tokio_file.rs:16` | `Mutex<File>` | 串行化所有 I/O | positioned I/O / RwLock 回退 |
| `winio.rs:29` | `Mutex<File>` | 同上 | IOCP OVERLAPPED |
| `connection.rs:34` | `Mutex<HashMap>` | 256 连接竞争 | `DashMap` |
| `buffer.rs:15` | `std::sync::Mutex<VecDeque>` | 阻塞+毒化恢复 | `crossbeam::ArrayQueue` |
| `resources.rs:37` | `std::sync::Mutex<HashMap>` | 阻塞+毒化恢复 | `tokio::sync::RwLock` |
| `ftp.rs:57` | `Mutex<ConnectionState>` | 只读操作也排他 | `RwLock` 或分离字段 |

### 12.2 AppState 锁碎片化

**现状**：6 个 `Arc<Mutex<...>>` 字段，`task_fn` 每迭代锁 5 次，`cancel_task` 嵌套锁 tasks→handles（死锁风险）。

**方案**：
1. `tasks` 进度改为 `watch` 通道（见 16.3）
2. `handles` 改为 `DashMap<String, JoinHandle<()>>`
3. 嵌套锁：先锁 `handles` abort，再锁 `tasks` 更新状态，顺序固定防死锁

### 12.3 暂停检测轮询 → 即时唤醒

**现状**：200ms 轮询 × 1500 次 = 5 分钟超时，每次都锁 `tasks`。

**方案**：每任务持有一个 `watch::Receiver<bool>`，pause 设为 true 时 `changed().await` 即时返回。

## 十三、类型安全与 API 设计

### 13.1 TaskInfo.status 应为 DownloadState 枚举

**现状**：`pub status: String`，比较用 `task.status == "paused"`，拼写错误静默失败。

**方案**：改为 `pub status: DownloadState`，移除 `status` 模块字符串常量，所有比较改为枚举 match。

### 13.2 ProtocolKind / StorageKind 枚举应改为 Arc<dyn Trait>

**现状**：每新增协议/存储后端需改枚举 + N 处 match。测试代码 `Mock` 变体用 `#[cfg(test)]` 混入生产枚举。

**方案**：改为 `Arc<dyn Protocol>` / `Arc<dyn AsyncStorage>`。`AsyncStorage` 已 object-safe，`Protocol` 需移除 RPITIT 或改为 `Pin<Box<dyn Future>>`。

### 13.3 VerifierKind 单变体枚举应简化

**现状**：仅有 `Cpu(CpuVerifier)` 变体，手动 `Clone` impl 即为 Bug #2。

**方案**：GPU verifier 功能实现前直接用 `CpuVerifier`，删除 `VerifierKind` 枚举。待 GPU 可用后再引入。

### 13.4 重复类型定义消除

| 重复项 | 位置 | 方案 |
|--------|------|------|
| `SnifferResource` 两套定义 | `commands.rs:94` vs `resources.rs:16` | 统一为一个，app 层直接用 sniffer 层的类型 |
| `ConnectionConfig` vs `PoolConfig` | `config.rs:41` vs `connection.rs:12` | 统一为一个 |
| `dirs()` 函数 | `config.rs:94` vs `commands.rs:172` | 保留 qf-core 一份，qf-app re-export |
| `user_agent` 重复定义 | `config.rs:34` vs `http.rs:24` | 提取为 `qf-core::USER_AGENT` 常量 |

### 13.5 类型安全增强（newtype）

| 类型 | 当前 | 目标 |
|------|------|------|
| `Etag` | `Option<String>` | `Etag(String)` newtype，强制引号格式 |
| `LastModified` | `Option<String>` | `LastModified(DateTime<Utc>)`，可比较 |
| `ContentType` | `Option<String>` | `ContentType(String)` newtype，可解析 MIME |
| `NodeId` | `[u8; 20]` | `NodeId([u8; 20])` newtype，验证非零 |
| `DownloadSource` key | `String` | `SourceUrl(String)` / `PeerAddr(String)` newtype |
| `PeerScore.stability` | `f64` | 私有字段 + setter 验证 `[0.0, 1.0]` |

### 13.6 DownloadConfig Builder 模式

**现状**：7 个字段，多数有非显然默认值。`DownloadConfig { field1: val1, ..Default::default() }` 不验证字段间约束（如 `max_concurrent_fragments` 不应超过 `max_global_connections`）。

**方案**：引入 `DownloadConfig::builder()` → `DownloadConfigBuilder`，setter 带验证逻辑。

## 十四、前端-后端契约优化

### 14.1 TaskInfo 摘要与详情分离

**现状**：`get_task_list()` 每次返回全部 TaskInfo（8 个 String 字段），2 秒轮询，100 任务时 ~10KB JSON/次。

**方案**：拆为两种响应：
```rust
struct TaskSummary { id, status: DownloadState, progress, speed }  // 轮询用
struct TaskDetail { // 全部字段，按需请求 }
```
轮询负载降低 ~60%。

### 14.2 DownloadProgress 与 TaskInfo 合并

**现状**：两套结构体 7/9 字段重叠，前端需处理两种形状。

**方案**：统一为 `TaskInfo`，通过 Tauri 事件推送增量更新。

### 14.3 SnifferResource 元数据丢失

**现状**：qf-sniffer 版有 `file_size`、`content_type`、`source_page`，qf-app 版缺少这些字段。

**方案**：qf-app 直接 re-export qf-sniffer 的 `SnifferResource`，删除重复定义。

### 14.4 结构化错误类型

**现状**：所有命令返回 `Result<T, String>`，前端无法区分错误类型。

**方案**：
```rust
#[derive(Debug, thiserror::Error, serde::Serialize)]
enum AppError {
    #[error("任务不存在: {0}")] TaskNotFound(String),
    #[error("任务已存在: {0}")] TaskAlreadyExists(String),
    #[error("网络错误: {0}")] Network(String),
    #[error("配置错误: {0}")] Config(String),
}
```

### 14.5 create_task 返回值不足

**现状**：仅返回 task_id，前端需轮询等元数据出现。

**方案**：返回 `CreateTaskResponse { id, metadata: Option<FileMetadata> }`，probe 完成后立即可用。

### 14.6 probe_metadata 创建新 reqwest::Client

**现状**：每次探测创建新 Client，无连接复用、无 TLS session 缓存。

**方案**：AppState 持有一个 `Arc<reqwest::Client>`，所有探测共用。

## 十五、内存分配热路径优化

### 15.1 QUIC 请求构造优化

**现状**：每次请求 parse URL + format String + to Vec<u8> = 3 次堆分配。

**方案**：预分配请求模板 buffer，仅替换 URL 路径部分。

### 15.2 QUIC 响应头处理优化

**现状**：`key.trim().to_ascii_lowercase()` 每行分配新 String。

**方案**：用 `eq_ignore_ascii_case` 比较而不分配。

### 15.3 分片执行减少 clone

**现状**：`self.url.clone()`（String 完整拷贝）、`self.storage.clone()`、`self.protocol.clone()`。

**方案**：URL 改为 `Arc<str>`，protocol/storage 已是 Arc 内部，直接 clone Arc。

### 15.4 fragment_infos() 避免全量 clone

**现状**：返回 `Vec<FragmentInfo>`，每元素 clone 含 `Option<String>`。

**方案**：返回 `&[FragmentInfo]` 或 `Arc<[FragmentInfo]>`。

### 15.5 get_task_list 减少 clone

**现状**：每 2 秒 clone 全部 TaskInfo。

**方案**：配合 watch channel，前端 `changed()` 后仅获取差异。

### 15.6 BandwidthTracker 清理

**现状**：`samples: VecDeque<u64>` 存 1000 条采样但 EWMA 仅需前值。

**方案**：移除 `samples` 字段，仅保留 `ewma`、`alpha`、`sample_count: u64`。

### 15.7 parse_content_disposition 减少 Vec 分配

**现状**：`splitn(3, '\'').collect::<Vec<&str>>()` 仅检查 `len() == 3`。

**方案**：用 iterator `.nth()` 逐步消费，避免 Vec 分配。

### 15.8 qf-app 测试 inner 函数 200+ 行重复

**现状**：`commands.rs:868-1099` 与 Tauri 命令函数仅参数类型不同（`&AppState` vs `tauri::State`），行为可能分化。

**方案**：抽取通用实现函数 `fn do_create_task(state: &AppState, ...) -> Result<...>`，Tauri 命令和测试函数都调用它。
